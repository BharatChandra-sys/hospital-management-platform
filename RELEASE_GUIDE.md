# Release Guide

This guide explains how to create and manage releases for the Hospital Management Platform.

## Release Process

### 1. Automated Release (Recommended)

The project includes automated release workflows that create professional releases with proper changelogs.

#### Create Release via GitHub UI

1. **Go to Releases**
   - Navigate to your repository on GitHub
   - Click on "Releases" in the right sidebar
   - Click "Create a new release"

2. **Configure Release**
   - **Tag**: `v1.0.0` (follow semantic versioning)
   - **Title**: `Hospital Management Platform v1.0.0`
   - **Description**: Use the template from `.github/RELEASE_TEMPLATE.md`

3. **Publish Release**
   - Check "Set as the latest release"
   - Click "Publish release"

#### Create Release via GitHub Actions

1. **Trigger Workflow**
   - Go to "Actions" tab in your repository
   - Click "Create Release" workflow
   - Click "Run workflow"
   - Enter version (e.g., `v1.0.0`)
   - Click "Run workflow"

2. **Automated Process**
   - Runs all tests
   - Builds both frontend and backend
   - Creates GitHub release with detailed notes
   - Updates CHANGELOG.md automatically

### 2. Manual Release Process

#### Prerequisites
```bash
# Install GitHub CLI
# Windows
winget install GitHub.cli

# Verify installation
gh --version
```

#### Create Release Manually
```bash
# 1. Create and push tag
git tag -a v1.0.0 -m "Release version 1.0.0"
git push origin v1.0.0

# 2. Create release with GitHub CLI
gh release create v1.0.0 \
  --title "Hospital Management Platform v1.0.0" \
  --notes-file .github/RELEASE_TEMPLATE.md \
  --latest

# 3. Upload additional assets (optional)
gh release upload v1.0.0 ./documentation.pdf
```

## Version Numbering

Follow [Semantic Versioning](https://semver.org/):

- **MAJOR** (1.x.x): Breaking changes
- **MINOR** (x.1.x): New features, backward compatible
- **PATCH** (x.x.1): Bug fixes, backward compatible

### Examples
- `v1.0.0` - Initial release
- `v1.1.0` - Added notifications system
- `v1.1.1` - Fixed authentication bug
- `v2.0.0` - Major API changes

## Release Checklist

### Pre-Release

- [ ] **Code Quality**
  - All tests passing
  - Code reviewed and approved
  - Security scan completed
  - Documentation updated

- [ ] **Version Updates**
  - Update version in `frontend/package.json`
  - Update version in `backend/requirements.txt` comments
  - Update CHANGELOG.md with new version

- [ ] **Testing**
  - Manual testing completed
  - Deployment tested on staging
  - Performance benchmarks verified
  - Security testing completed

- [ ] **Documentation**
  - README.md updated
  - API documentation current
  - Deployment guides verified
  - Breaking changes documented

### Release Creation

- [ ] **GitHub Release**
  - Tag created with proper version
  - Release notes comprehensive
  - Assets uploaded if needed
  - Latest release marked

- [ ] **Deployment**
  - Production deployment verified
  - Database migrations run
  - Environment variables updated
  - SSL certificates valid

### Post-Release

- [ ] **Communication**
  - Release announcement posted
  - Documentation sites updated
  - Community notified
  - Social media updated

- [ ] **Monitoring**
  - Error tracking active
  - Performance monitoring
  - User feedback collected
  - Metrics dashboard updated

## Release Assets

2. **Documentation**
   - User manual (PDF)
   - API documentation export

   - Installer packages


## Rollback Procedure

If issues are found after release:

### 1. Immediate Response
```bash
# Mark release as pre-release
gh release edit v1.0.0 --prerelease

# Or delete if critical
gh release delete v1.0.0 --yes
```

### 2. Fix and Re-release
```bash
# Create hotfix branch
git checkout -b hotfix/v1.0.1

# Make fixes and test
# ... fix code ...

# Create new release
git tag -a v1.0.1 -m "Hotfix release"
git push origin v1.0.1
gh release create v1.0.1 --latest
```

## Best Practices

### Do's ✅
- Always test before releasing
- Follow semantic versioning strictly
- Provide detailed release notes
- Include migration guides for breaking changes
- Monitor releases after deployment
- Maintain backwards compatibility when possible

### Don'ts ❌
- Don't release untested code
- Don't skip version numbers
- Don't release without proper documentation
- Don't ignore security vulnerabilities
- Don't release during peak hours without planning
